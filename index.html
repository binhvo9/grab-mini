Grab Mini — Ride-Hailing System Design

A condensed architecture showcase: rider request → driver match → live trip tracking → fare settlement. Read the full deep dive (trade-offs, failure modes, capacity estimation) in README.md.

1. Sequence Diagram — Ride Request to Trip Completion

Rider requests a ride, the matching service finds the nearest driver via the geo index, and live location updates stream back to the rider during the trip.

sequenceDiagram
    actor Rider
    actor Driver
    participant Gateway as API Gateway
    participant Matching as Matching Service
    participant Pricing as Pricing Service
    participant Trip as Trip Service
    participant Broker as Kafka (Event Bus)
    participant Loc as Location Service (Redis Geo)
    participant Notif as Notification Service

    Rider->>Gateway: POST /ride-requests (pickup, dropoff)
    Gateway->>Pricing: Estimate fare
    Pricing-->>Gateway: Fare estimate
    Gateway->>Matching: Find nearby drivers
    Matching->>Loc: GEOSEARCH nearby drivers
    Loc-->>Matching: candidate driver list
    Matching->>Broker: publish RideRequested
    Broker->>Notif: consume RideRequested
    Notif->>Driver: Push notification (new ride nearby)
    Driver->>Gateway: POST /rides/{id}/accept
    Gateway->>Trip: Create trip record
    Trip->>Broker: publish TripStarted
    Broker->>Notif: consume TripStarted
    Notif->>Rider: Push notification (driver assigned)

    loop every 4s during trip
        Driver->>Loc: Update GPS location
        Loc->>Broker: publish LocationUpdated
        Broker->>Notif: stream to Rider via WebSocket
    end

    Driver->>Gateway: POST /rides/{id}/complete
    Gateway->>Trip: Mark trip completed
    Trip->>Pricing: Calculate final fare
    Trip->>Broker: publish TripCompleted
In plain English: You tap "Book ride" → the app instantly quotes a price and shouts "who's nearby?" → the closest driver gets a notification and taps "Accept" → you're both notified "matched". While the trip runs, the driver's phone pings its location every 4 seconds so your map updates live. When the driver taps "Complete", the app calculates the final price and closes the trip. Every step is just a message passed to the next person/system — like a relay race.

2. Functional / Component Architecture

How core services connect: client apps talk to an API gateway, services publish events onto Kafka, and a Flink speed layer + S3 batch layer (Lambda Architecture) feed analytics.

flowchart TB
    subgraph Clients
        RiderApp[Rider App]
        DriverApp[Driver App]
    end

    RiderApp --> GW[API Gateway / BFF]
    DriverApp --> GW

    GW --> Matching[Matching Service]
    GW --> Pricing[Pricing Service]
    GW --> Trip[Trip Service]
    GW --> Auth[Auth Service]

    Matching --> GeoIndex[(Redis Geo Index)]
    Trip --> TripDB[(PostgreSQL: Trips and Users)]
    Pricing --> TripDB

    Matching -- publish --> Bus[[Kafka Event Bus]]
    Trip -- publish --> Bus
    DriverApp -- location ping --> Bus

    Bus --> Notif[Notification Service]
    Bus --> StreamProc[Stream Processor - Flink Speed Layer]
    Bus --> BatchSink[Batch Sink - S3 Data Lake]

    Notif --> RiderApp
    Notif --> DriverApp
    StreamProc --> GeoIndex
    StreamProc --> Analytics[(Real-time Dashboards)]
    BatchSink --> BatchProc[Batch Layer - Spark / Athena]
    BatchProc --> Serving[(Serving Layer - Analytics DB)]
In plain English: This is the "org chart" of the system — who talks to whom. Both apps walk through one front door (API Gateway). Behind that door, specialist teams each do ONE job — Matching finds drivers, Pricing does fares, Trip keeps records. Instead of calling each other directly for every update, they pin notes to one shared notice board (Kafka). Anyone interested — notifications, live dashboards, or end-of-day reporting — reads the board at their own pace, so one slow team never jams up everyone else.

3. Final Architecture (Concrete Stack)

Same shape, with concrete technology choices: Kong/Envoy gateway, Kafka with replication factor 3, Flink for real-time processing, and Spark over an S3 data lake for batch analytics.

flowchart LR
    User[Rider/Driver Mobile Apps] --> CDN[CDN - static assets]
    User --> LB[Load Balancer]
    LB --> APIGW[API Gateway - Kong/Envoy]

    APIGW --> SvcMatching[Matching Service]
    APIGW --> SvcPricing[Pricing Service]
    APIGW --> SvcTrip[Trip Service]

    SvcMatching <--> Redis[(Redis Geo Cluster)]
    SvcTrip <--> PG[(PostgreSQL - Multi-AZ)]

    SvcMatching -- produce --> Kafka[[Apache Kafka - RF=3]]
    SvcTrip -- produce --> Kafka
    DriverGPS[Driver GPS Stream] -- produce --> Kafka

    Kafka --> Flink[Apache Flink - Speed Layer]
    Kafka --> S3[(S3 Data Lake)]

    Flink --> Redis
    Flink --> RTDash[Real-time Dashboard]

    S3 --> Spark[Spark Batch Jobs]
    Spark --> Warehouse[(Data Warehouse)]
    Warehouse --> RTDash
In plain English: Same picture, with real product names. Kong/Envoy is the front-door bouncer. Kafka is the notice board — but now with 3 copies of every note pinned in 3 different rooms so nothing's lost if one floods. Redis is a super-fast phonebook answering "which drivers are near here right now?". PostgreSQL is the permanent filing cabinet for trips and payments — never allowed to disappear. Flink reacts to the notice board instantly (live maps/dashboards), while Spark processes yesterday's notes overnight for management reports. Same data, two speeds, two jobs.
